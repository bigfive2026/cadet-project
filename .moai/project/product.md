# Product — ArtBridge

> 창작자(신진작가)와 팬을 잇는 후원 커뮤니티 MVP
> 문서 기반: `0. 문서 목적.md`(ArtBridge PRD v0.2). 코드 식별자는 `ArtBridge`, 사용자 노출명은 확정 후 find-replace.

---

## 1. 제품 개요

### 1.1 한 줄 정의

ArtBridge는 신진작가가 자신의 **창작 스튜디오**를 개설하고, 팬이 **멤버십·유료 포스트·클럽/프로그램**을 통해 작가를 후원하고 교류할 수 있는 **창작자-팬 후원 커뮤니티**다.

### 1.2 제품 콘셉트

Steadio 벤치마크. 핵심 기능 축을 과제 기간 안에 구현 가능한 수준으로 추려 다음 흐름으로 구성한다.

```
작가 스튜디오 → 멤버십 → 유료/멤버 전용 포스트 → 클럽/프로그램
→ 팬 참여 신청 → 수락/결제 → 커뮤니티 → 리뷰
```

### 1.3 RFP 도메인 변환 (배경)

원본 RFP는 의뢰인↔프리랜서 외주 매칭(WorkBridge) 플랫폼이다. ArtBridge는 양면형 거래 플랫폼 **구조는 유지**하되 도메인을 창작자-팬 커뮤니티로 변환했다.

| RFP 구조 | ArtBridge 변환 |
|---|---|
| 의뢰인 | 팬 / 후원자 / 참여자 |
| 프리랜서 | 크리에이터 / 신진작가 |
| 프로젝트 | 클럽 / 프로그램 / 멤버십 오퍼링 |
| 지원 | 팬의 참여 신청 또는 멤버십 가입 |
| 지원 수락/거절 | 크리에이터의 신청 승인/거절 |
| AI 단가 분석 | AI 가격·혜택·프로그램 구성 추천 |
| 계약 | 프로그램 참여 계약서 / 멤버십 약관 |
| 결제/에스크로 | 팬 선결제 → 플랫폼 보관 → 완료 후 정산 |
| 리뷰 | 팬 후기 + 크리에이터/팬 상호 평가 |
| 알림 | 신청/수락/거절/마감/결제·리뷰 요청 알림 |

## 2. 핵심 문제와 사용자

### 2.1 문제 정의

- **크리에이터**: 팬과 소통할 채널이 흩어져 있고, 지속 수익을 만들기 어렵다. SNS는 노출·홍보에 좋지만 멤버십·유료 콘텐츠·커뮤니티·프로그램 운영을 한곳에서 관리하기 어렵다.
- **팬**: 좋아하는 작가를 후원하고 더 깊게 연결되고 싶지만, 후원·콘텐츠 구매·프로그램 참여·커뮤니티 활동이 분산되어 지속적 관계를 맺기 어렵다.

ArtBridge는 작가의 창작 활동을 하나의 스튜디오에 모으고, 팬이 멤버십·유료 포스트·클럽/프로그램으로 참여할 수 있게 만들어 **지속 가능한 수익화와 커뮤니티 빌딩**을 돕는다.

### 2.2 사용자 유형

**Creator / Artist (크리에이터)** — 신진작가, 일러스트레이터, 에세이스트, 독립출판 작가, 뮤지션, 공예 작가 등.
주요 행동: 스튜디오 페이지 생성 · 자기소개/링크/포트폴리오 등록 · 멤버십 플랜 개설 · 유료/멤버 전용 포스트 발행 · 클럽/챌린지/클래스 개설 · 팬 신청 수락/거절 · 커뮤니티 소통 · 완료 후 리뷰 확인.

**Fan (팬)** — 좋아하는 작가를 후원하고, 콘텐츠를 구매하거나 프로그램에 참여하려는 사용자.
주요 행동: 작가·프로그램 탐색 · 멤버십 가입 · 멤버 전용 포스트 확인 · 클럽/프로그램 참여 신청 · 결제·계약 동의 · 커뮤니티 글 작성 · 완료 후 리뷰 작성.

---

## 3. 핵심 가치 제안

크리에이터의 **지속 가능한 창작 수익화**와 팬의 **깊이 있는 참여**를 하나의 스튜디오에서 해결한다. 거래 신뢰는 선결제→보관→정산 상태머신과 상호 리뷰로 보증한다(실제 송금 없이 상태 시뮬레이션).

## 4. MVP 범위 (MoSCoW)

### 4.1 P0 — 반드시 구현

| 기능 | 설명 | 구현 원칙 |
|---|---|---|
| 역할 기반 로그인 | Creator/Fan 접속 | 가능한 Auth 활용 |
| 크리에이터 스튜디오 | 작가 소개, 이미지, 링크, 탭 UI | 시드 데이터 필수 |
| 멤버십 플랜 | 가격, 혜택, 가입 CTA | 단일 플랜부터 허용 |
| 포스트 | PUBLIC / MEMBERS_ONLY / PAID 구분 | PAID는 Mock 구매 또는 잠금 UI |
| 클럽/프로그램 CRUD | 제목, 설명, 가격, 기간, 모집 인원 | RFP의 Project 대응 |
| 팬 신청 | 프로그램 참여 신청 | 중복 신청 방지 |
| 수락/거절 | 크리에이터 신청 처리 | 수락 시 타 대기 신청 자동 거절 옵션 |
| 알림 | 신청/수락/거절/자동거절/마감 | 인앱 알림만 |
| 계약/약관 | 프로그램 참여 계약서 생성 | 체크박스 서명 허용 |
| 결제 | Mock 우선, sandbox adapter 선택 | 실제 결제 금지 |
| 커뮤니티 | 멤버/참여자 전용 글 목록/작성 | 댓글은 선택 |
| 리뷰 | 완료 후 별점/텍스트 리뷰 | 1회 작성, 수정 불가 |
| 배포 | Vercel 등 외부 URL | 시드 데이터 포함 |

### 4.2 P1 — 시간이 되면 구현

- AI 가격·혜택 추천 (프로그램 설명 → 가격/혜택/운영 구성 추천)
- 단건 유료 포스트 구매 Mock
- 프로그램 검색/필터 (카테고리, 가격, 모집 상태)
- 멤버 관리 상세 (활동 이력 요약)
- 추천 프로그램 (같은 카테고리/랜덤)
- 이미지 업로드 (Cloudinary/Supabase Storage 또는 URL 입력)

### 4.3 Won't — 이번 MVP 제외

| 기능 | 제외 이유 |
|---|---|
| 반복 구독 결제 | 구현 비용·장애 리스크 큼 |
| 실시간 채팅 | 발표 효과 대비 난도 높음 |
| 쿠폰 발행 | Steadio 확장 기능만 |
| 오디오 플레이어 | 핵심 흐름과 무관 |
| 배송지 수집 | 물리 상품 배송은 범위 밖 |
| 환불 API | 도전 과제 성격 |
| 관리자 정산 페이지 | DB 상태 전환으로 대체 |
| 다회차 가격 협상 | 기능 폭발 위험 |

## 5. 핵심 유저 플로우 (데모 최소 성공선)

1. 랜딩에서 가치 제안 표시
2. 크리에이터 로그인 → 스튜디오 확인/수정
3. 멤버십 플랜 생성/확인
4. 멤버 전용 포스트 작성
5. 클럽/프로그램 생성
6. 팬으로 전환 로그인
7. 팬이 스튜디오·프로그램 탐색
8. 팬이 클럽/프로그램 참여 신청
9. 크리에이터가 신청 수락
10. 팬에게 수락 알림 표시
11. 팬이 계약/약관 동의 후 결제
12. 결제 완료 후 멤버/참여자 콘텐츠·커뮤니티 접근
13. 크리에이터가 멤버/참여자 목록 확인
14. 프로그램 완료 처리 후 리뷰 작성

> 시간 부족 시 최소 8개만 살아 있으면 제출 가능 MVP: Mock/실제 로그인 역할 전환 · 스튜디오 표시 · 멤버십 플랜 표시 · 멤버 전용 포스트 잠금/공개 · 클럽/프로그램 생성·상세 · 팬 참여 신청 · 수락/거절+알림 · 결제 Mock/sandbox 상태 전환 후 리뷰.

## 8. 제품 원칙

이번 프로젝트에서 좋은 기능은 "멋진 기능"이 아니라 **데모를 살리는 기능**이다. 판단이 애매하면:

1. 최종 데모 경로에 직접 필요한가?
2. 구현하지 않으면 발표가 무너지는가?
3. Mock으로 대체해도 비즈니스 의미가 전달되는가?
4. 오늘 안에 붙이고 검증할 수 있는가?
5. 배포 환경에서 안정적으로 동작할 가능성이 높은가?

하나라도 명확하지 않으면 만들지 않는다.

> **기능 Freeze**: 6/21 이후 새 기능 추가 없이 데모 안정화만 수행. 허용: 버그/타입/빌드 수정, 링크 연결, 빈 화면 제거, 시드 보강, UI polish, 배포 환경변수 정리. 금지: 실제 PG 신규 연동, Auth 전면 교체, 대규모 모델 변경, 채팅/쿠폰/오디오/배송지 추가.
